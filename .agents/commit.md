# 커밋 규칙

## 형식

```
<type>: <한 줄 요약>

<선택: 본문 — 왜 이렇게 했는가>
```

## type

- `feat` — 새 기능
- `fix` — 버그 수정
- `refactor` — 동작 변경 없이 구조 개선
- `test` — 테스트 추가/수정
- `docs` — 문서만 변경
- `chore` — 빌드·설정·메타
- `perf` — 성능 개선
- `style` — 포맷·공백·세미콜론

## 단위

- 1 task = 1 커밋
- 무관한 변경 섞지 않음
- 리팩토링 커밋과 기능 추가 커밋 분리

## 금지

- `main` 직접 커밋
- `--no-verify` 로 훅 우회
- 테스트 없는 기능 커밋
- "WIP" 커밋을 머지 브랜치에 남기기
- 여러 도메인 변경을 한 커밋에 묶기

## 좋은 예

```
feat: 인재 등록 폼 섹션별 파일 분리

14섹션 폼 제출 로직을 섹션 단위 파일로 나눠 테스트 표면을 확장.
orchestrator 는 Promise.allSettled 로 부분 실패 허용.
```
